# BioDraw 编码约束与提交规范 v1

## 1. 文档目的

本文档用于统一 BioDraw 第一阶段开发过程中的编码约束、模块职责边界、提交原则和验收前检查项，以降低智能体和人工开发同时参与时的混乱风险。

---

## 2. 总体原则

1. **先类型，后实现**
2. **先主链路，后扩展能力**
3. **先模块职责清晰，后谈复用**
4. **尽量做最小可工作的改动**
5. **不在未定文档处自行发明复杂机制**
6. **所有核心业务数据优先落在 domain**
7. **React 组件尽量只做 UI 与事件编排，不堆积大量业务规则**

---

## 3. 模块职责约束

## 3.1 domain
- 负责类型定义
- 负责纯函数与业务规则
- 不允许依赖 React
- 不允许依赖 Konva
- 不允许依赖 DOM

## 3.2 render
- 负责把领域数据映射成画面
- 可依赖 Konva
- 不应成为最终业务规则定义层

## 3.3 features
- 负责用户交互入口和界面组织
- 不直接手写复杂 pathData 算法
- 不直接定义核心领域语义

## 3.4 state
- 负责应用状态容器
- 不应把所有业务逻辑直接塞进 reducer 或 context

## 3.5 infrastructure
- 负责外部存储、素材加载、导出服务等
- 不定义核心业务模型含义

---

## 4. 文件放置约束

### 类型定义
优先放在：
```text
src/domain/*/types.ts
```

### 纯业务逻辑
优先放在：
```text
src/domain/*/*.ts
```

### React 组件
优先放在：
```text
src/features/*
src/render/*
src/pages/*
```

### 工具函数
若与业务强相关，不放 shared，应放在对应 domain 下。  
只有真正通用的小工具才放到：
```text
src/shared/utils/
```

---

## 5. 命名规范

## 5.1 文件命名
建议：
- React 组件：`PascalCase.tsx`
- 普通逻辑文件：`camelCase.ts` 或明确名词式文件名
- 类型文件：`types.ts`

## 5.2 类型命名
- 类型名使用 `PascalCase`
- 枚举值或字面量使用明确英文标识

## 5.3 变量命名
- 避免 `data`, `item`, `temp` 这种无意义泛名滥用
- 尽量体现其领域含义，例如：
  - `selectedObjectIds`
  - `currentTimeMs`
  - `materialCatalog`

---

## 6. 状态修改约束

1. 不要在多个组件里散落修改同一份关键状态
2. 关键状态变更优先通过明确 action 或 mutation 完成
3. 对象、路径、动画的更新逻辑尽量集中
4. 不要让 Konva 事件回调直接写一堆杂乱更新逻辑

---

## 7. 代码实现顺序约束

实现一个新能力时，建议遵循：

1. 补类型
2. 补 domain 纯逻辑
3. 补 state 接入
4. 补 render 层
5. 补 features 交互入口
6. 最后补样式和细节

不要反过来先把 UI 写得很热闹，再倒推数据模型。

---

## 8. 提交原则

### 8.1 一次提交尽量只解决一个明确问题
例如：
- “接入 Konva Stage”
- “实现素材拖拽进入画布”
- “实现对象选中与拖拽”

### 8.2 避免超大杂糅提交
不要一次提交同时包含：
- 新目录结构
- 新类型系统
- 新播放器
- 新素材库逻辑
- 新样式大改

这样会让回退和审查都很困难。

### 8.3 优先最小可工作改动
如果一个目标可以通过较小改动达成，优先较小改动。

---

## 9. 每次改动前建议自检

1. 这个改动属于当前里程碑吗
2. 这个改动是否需要先补文档
3. 这个改动是否会把简单问题做复杂
4. 这个改动是否污染了模块边界
5. 这个改动是否破坏主链路稳定性

---

## 10. 每次改动后建议检查

1. 是否仍能正常运行
2. 当前阶段验收点是否成立
3. 新增文件是否放在正确目录
4. 是否引入了不必要的抽象
5. 是否新增了文档未定义的复杂交互
6. 是否存在明显的死代码或重复逻辑

---

## 11. 智能体参与编码的特别约束

1. 不自行扩展未在文档中确认的大功能
2. 不擅自回退到旧设计思路
3. 不把“未来可能会用到”作为当前复杂实现理由
4. 不把大量业务逻辑塞进 UI 组件
5. 对不确定的设计边界，应优先遵循现有文档，而不是自行发挥

---

## 12. 第一阶段推荐验收顺序

每次提交或阶段结束后，优先检查：

1. 页面能否打开
2. 当前链路是否可操作
3. 交互是否符合文档
4. 状态是否同步
5. 是否存在明显阻断性问题

---

## 13. 结语

这份编码约束与提交规范的目标，是为 BioDraw 当前阶段建立一套**可收敛、可审查、可协作**的开发纪律。

后续所有编码和提交，都应尽量反复对照下面这句话：

**当前这次改动，是否真的在帮助项目更稳定地推进第一阶段主链路？**

如果没有，就不应优先合入。
