# BioDraw 详细设计方案

## 1. 文档说明

本文档用于整理当前阶段已经明确的产品方向、核心模块设计思路与关键技术路线，作为 BioDraw 项目设计、开发的当前基线文档。

本文档的目标不是覆盖最终全部细节，而是先建立一份足够清晰、能够指导后续开发工作的设计基线。

---

## 2. 产品定位

## 2.1 产品一句话定义

**BioDraw 是一款面向中学生物教师的动态示意图制作工具。它以现成生物教学素材为基础，帮助教师通过“选素材—搭场景—设动作—预览—导出”的流程，快速制作适合课堂演示的动态示意图。**

## 2.2 目标用户

目标用户主要是：

- 中学生物教师
- 需要在备课或课堂中演示某些生物过程的教师
- 不具备专业设计或动画制作背景，但希望低门槛完成教学动画表达的用户

## 2.3 核心目标

本产品不是要做一个通用绘图软件，也不是要做一个复杂的专业动画软件，而是要解决以下问题：

- 让老师更容易制作生物动态示意图
- 让老师制作出来的内容更适合课堂讲解
- 让学生更容易理解抽象的生物过程
- 让教学动画成果能够方便地复用到 PPT、浏览器等常见场景

## 2.4 产品设计原则

后续所有设计与开发决策，应尽量服从以下原则：

1. **优先服务教学表达，而不是通用编辑能力**
2. **优先降低教师使用门槛，而不是增加专业设计能力**
3. **优先保障课堂使用稳定性，而不是追求复杂炫酷效果**
4. **优先让老师“选素材并组织过程”，而不是“从零绘制一切”**
5. **优先构建完整主链路，而不是过早追求大而全**

---

## 3. 对旧版本的判断

当前旧版本应视为原型阶段的试错结果，其主要价值在于帮助确认：

**这个产品不应沿着“通用绘图编辑器”的方向继续发展。**

旧版本的主要偏差包括：

- 过于强调对象属性编辑
- 交互重心偏向坐标、尺寸、层级等底层编辑
- 界面结构更像通用编辑器，而不像教学动画工具
- 素材、动画设计、导出展示这三条主线没有成为真正主角

因此，后续不应继续围绕旧版结构修修补补，而应以新的产品定义重新设计。

---

## 4. 用户核心使用流程

教师从产生需求到完成成果的标准流程，应定义为：

## 4.1 查找素材

用户根据教学内容，在素材库中查找所需素材。

## 4.2 拖入画布

用户将素材直接拖放到动画制作画布中，开始搭建初始场景。

## 4.3 场景布置与动画设计

用户在画布中完成以下工作：

- 自由拖拽元素到目标位置
- 通过坐标输入进行精细调整（作为辅助方式保留）
- 为元素设计运动轨迹
- 设置元素在不同时间的移动、形态变化、样式变化
- 规划速度、时长和先后顺序

这一阶段的核心表达可概括为：

**什么元素，在什么时间，到什么位置，变成什么样子。**

## 4.4 播放预览

用户可直接播放动画，检查效果是否满足教学需要。

## 4.5 调整修改

用户根据预览结果继续修改场景与动画。

## 4.6 导出成果

用户将完成的成果导出，用于：

- 插入 PPT 播放
- 电脑本地播放
- 浏览器直接播放
- 其他教学场景复用

---

## 5. 产品核心模块

当前版本的核心功能结构，应明确分为三大模块：

1. **素材**
2. **画布动画设计**
3. **结果保存与导出**

后续软件的主界面结构、数据模型和技术选型，都应围绕这三块展开。

---

# 6. 模块一：素材

## 6.1 模块定位

素材模块的核心原则是：

**尽量提供现成可用的生物教学素材，让老师优先“选素材”，而不是“画素材”。**

用户自定义制作素材的能力可以保留，但不应作为第一阶段重点，应视为中后期扩展能力。

## 6.2 素材形式结论

当前结论：

**主素材格式优先采用 SVG 矢量素材，PNG / WebP 作为补充。**

不建议以普通位图图片（PNG / JPG）作为核心素材体系。

## 6.3 为什么主选 SVG

SVG 更符合本项目需求，主要原因如下：

- 可无损缩放，适合课堂展示场景
- 便于统一风格
- 易于修改颜色、线条和局部结构
- 更适合程序控制与动画扩展
- 适合 Web 项目使用
- 可与后续轨迹、导出、浏览器播放形成更顺畅的技术链路

## 6.4 素材体系建议

建议构建分层素材体系。

### 第一层：核心教学素材（SVG 为主）

例如：

- 细胞
- 细胞膜
- 分子
- 蛋白通道
- 染色体
- 囊泡
- 箭头
- 标注框

### 第二层：辅助场景素材（PNG / WebP 补充）

例如：

- 背景
- 纹理
- 装饰性素材
- 暂未矢量化的素材

### 第三层：参数化高级素材（后期扩展）

例如：

- 可拉伸细胞膜
- 可调数量的分子群
- 可切换状态的蛋白通道
- 可配置路径的运动对象

## 6.5 素材获取策略

建议组合使用以下方式：

- 直接使用开放授权的现成 SVG 素材
- 从公开资源中收集参考素材，再进行统一风格化整理
- 对高频核心素材自行重绘或规范化处理

## 6.6 素材入库要求

素材不应只是一个文件，而应包含元数据。建议每个素材至少记录：

- `id`
- `名称`
- `分类`
- `子分类`
- `关键词`
- `默认尺寸`
- `推荐用途`
- `授权信息`
- `来源信息`
- `是否支持变色`
- `是否支持局部动画`
- `是否存在多个状态`

## 6.7 素材库分类体系（初稿）

素材库不应只是把文件堆在一起，而应形成适合教师快速查找和快速使用的层级结构。

### 设计原则

- 分类尽量贴近中学生物教学内容
- 一级分类不宜过多，便于快速定位
- 二级分类更贴近具体素材类型
- 支持分类浏览与关键词搜索并存
- 支持“常用”“最近使用”“收藏”等高频入口

### 一级分类建议

建议第一阶段采用以下一级分类：

1. 细胞与细胞器
2. 膜与运输结构
3. 分子与颗粒
4. 遗传与分裂
5. 组织与器官局部

建议优先顺序：

1. 跨膜运输
2. 有丝分裂 / 减数分裂
3. DNA 复制与表达
4. 光合作用与呼吸作用

其中第一批优先推荐从**跨膜运输**开始。

# 7. 模块二：画布动画设计

## 7.1 模块定位

画布动画设计模块是本产品的核心价值模块。

其目标不是做复杂通用动画，而是支持教师完成以下工作：

- 在画布中搭建初始场景
- 自由拖拽元素到合适位置
- 通过坐标辅助进行精细调整
- 设计元素的移动轨迹
- 设置元素沿轨迹运动
- 设置移动速度和持续时间
- 设置元素在不同时间点的状态变化
- 进行播放预览与反复修改

## 7.2 技术总路线

当前建议采用以下总体技术路线：

**React + react-konva（编辑交互层） + SVG Path（轨迹数据格式） + 自定义时间/动画系统（播放调度层）**

GSAP MotionPath 可作为早期验证或局部辅助工具，但不应成为整个动画系统的核心依赖。

## 7.3 为什么采用 Konva / react-konva

Konva 更适合编辑器型 2D 画布应用，原因包括：

- 提供对象模型
- 支持拖拽、选中、事件处理
- 支持图层管理
- 适合 Canvas 交互式编辑场景
- 与 React 集成较成熟

## 7.4 轨迹技术路线

轨迹建议统一采用 **SVG Path** 表示。

原因：

- 表达能力成熟
- 可描述直线、折线、贝塞尔曲线等
- 便于在 Web 端处理
- 便于未来导出与复用
- 可作为内部标准路径格式

## 7.5 第一阶段轨迹编辑建议

第一阶段建议只支持以下三类轨迹：

- 直线
- 折线
- 贝塞尔曲线

轨迹编辑方式建议采用：

- 锚点编辑
- 控制点编辑
- 可视化拖拽调整

即先实现一个“简化版钢笔工具”，而不是追求过于复杂的路径系统。

## 7.6 动画系统建议

内部动画系统不建议单纯依赖外部动画库，而应自行维护核心播放逻辑。

建议内部动画以“时间驱动对象状态”的方式建模。动画动作可以抽象为：

- 移动
- 沿路径运动
- 出现 / 消失
- 放大 / 缩小
- 旋转
- 透明度变化
- 颜色变化
- 状态切换

其中，“沿路径运动”是第一阶段重点。

## 7.7 速度与时长的处理建议

对用户可暴露：

- 时长
- 速度（慢 / 中 / 快或滑杆）

内部应主要维护：

- `path`
- `duration`
- `easing`
- `startTime`
- `endTime`
- `progress`

不建议内部仅以“速度”作为唯一核心参数。

## 7.8 画布设计的产品要求

需要明确：

- 画布是“场景搭建与过程设计区”，不是单纯绘图区
- 位置输入是辅助方式，主方式仍应是直接拖拽
- 预览是主流程的一部分，不是附属功能
- 动画系统后续必须能支持录制导出视频

## 7.9 第一阶段动画设计目标

第一阶段只需实现：

- 元素拖拽
- 坐标微调
- 轨迹可视化设计
- 沿路径运动
- 时长设置
- 基础速度控制
- 播放 / 暂停 / 重播预览

不建议第一阶段做：

- 高级曲线编辑器
- 复杂物理效果
- 大量特效系统
- 过重的专业时间线系统

## 7.10 动画对象与轨迹的数据模型（初稿）

动画系统的数据模型应围绕“对象、轨迹、动作、时间”四个核心要素建立。

### 场景对象基础结构

```ts
type SceneObject = {
  id: string
  type: string
  name: string
  materialId: string
  x: number
  y: number
  width: number
  height: number
  rotation: number
  scaleX: number
  scaleY: number
  opacity: number
  visible: boolean
  zIndex: number
  anchorX?: number
  anchorY?: number
  style?: Record<string, unknown>
  state?: string
  animations: AnimationClip[]
}
```

说明：

- `materialId` 用于指向素材库中的基础素材
- `state` 用于表示该对象当前形态或状态，例如“打开 / 关闭”“结合 / 分离”
- `animations` 用于描述该对象参与的动画片段

### 轨迹对象结构

轨迹建议单独建模，而不是只嵌在动画动作里。

```ts
type MotionPath = {
  id: string
  name?: string
  kind: 'line' | 'polyline' | 'bezier'
  points: PathPoint[]
  pathData: string
  closed: boolean
}

type PathPoint = {
  x: number
  y: number
  inHandleX?: number
  inHandleY?: number
  outHandleX?: number
  outHandleY?: number
}
```

说明：

- `points` 用于编辑态
- `pathData` 用于播放态和导出态
- 第一阶段轨迹编辑的核心是维护 `points -> pathData` 的转换

### 动画动作结构

```ts
type AnimationClip = {
  id: string
  type: 'move' | 'moveAlongPath' | 'fade' | 'scale' | 'rotate' | 'stateChange'
  startTimeMs: number
  durationMs: number
  easing?: string
  payload: Record<string, unknown>
}
```

其中，“沿路径运动”动作建议至少支持：

```ts
type MoveAlongPathPayload = {
  pathId: string
  followPathAngle?: boolean
  speedMode?: 'duration' | 'speed'
  speed?: number
  loop?: boolean
}
```

### 对象状态切换建议

为支持后续教学场景中的形态变化，建议对象可拥有多个状态：

```ts
type ObjectStateVariant = {
  key: string
  label: string
  overrides?: {
    style?: Record<string, unknown>
    visibleParts?: string[]
    hiddenParts?: string[]
  }
}
```

例如：

- 通道蛋白：关闭 / 打开
- 囊泡：未融合 / 融合中 / 已融合
- 染色体：复制前 / 复制后

# 8. 模块三：结果保存与导出

## 8.1 模块定位

导出模块必须从课堂使用场景反推设计，而不是单纯提供一个技术上的“另存为”。

当前已经明确，教师最常见的目标场景是：

- 在 PPT 中播放动画
- 在浏览器中直接播放动画

## 8.2 导出主路线结论

当前结论：

**第一阶段导出主格式应为 MP4 视频。**

同时需要保存工程文件，用于后续继续编辑。

因此，第一阶段成果类型应为：

- **工程文件**（用于继续编辑）
- **MP4 视频**（用于 PPT、本地播放器、浏览器播放）

## 8.3 为什么优先 MP4

主要原因：

- 最符合教师实际使用方式
- 可直接插入 PPT
- 浏览器兼容性高
- 课堂使用稳定
- 技术路径清晰

## 8.4 编码格式建议

视频导出建议采用：

- 视频编码：H.264
- 音频编码：AAC
- 封装格式：MP4

## 8.5 浏览器播放策略

第一阶段浏览器播放不需要单独发明复杂方案，可直接播放导出的 MP4。

后续如有必要，可扩展支持“网页演示包导出”，例如：

- HTML
- 场景 JSON
- 资源文件
- 浏览器内置播放器

但这不应成为第一阶段重点。

## 8.6 不建议优先做的导出方向

第一阶段不建议优先投入：

- 可嵌入 PPT 的专有动画对象导出
- 复杂 HTML 动画嵌入 PPT 方案
- 以 GIF 作为主导出格式

GIF 可作为后续补充导出，但不应作为主导出方案。

## 8.7 导出实现原则

编辑器内部应维护可编辑的场景与动画状态；导出时再将动画渲染并录制为视频。

也就是说：

- 编辑阶段：对象级、可修改
- 导出阶段：渲染为标准视频成果

后续动画系统设计必须能支持逐帧或稳定时间驱动渲染，以保障视频导出的可实现性。

# 9. 主界面结构建议

基于当前产品方向，新版本主界面建议围绕三大模块构建，而不是沿用传统通用编辑器布局。

建议结构如下：

## 左侧：素材库

- 分类浏览
- 搜索
- 常用素材
- 最近使用素材

## 中间：画布

- 场景搭建
- 对象拖拽
- 轨迹编辑
- 动画预览主视区

## 右侧或下方：动画设计区

- 动画动作设置
- 轨迹参数
- 时间 / 时长设置
- 状态变化设置

## 顶部：工程与预览控制区

- 新建
- 保存
- 导出
- 播放
- 暂停
- 重播

说明：

对象属性面板仍可保留，但应从“主角”降为“辅助区”。

# 10. 第一阶段开发优先级

## 10.1 必须优先完成的主链路

第一阶段建议优先实现一条完整的最小可用链路：

1. 基础素材库
2. 素材拖入画布
3. 元素自由拖拽与坐标微调
4. 轨迹设计
5. 元素沿轨迹运动
6. 播放预览
7. 工程保存
8. MP4 导出
